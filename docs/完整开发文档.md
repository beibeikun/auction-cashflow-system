# auction-app 完整开发文档

本文档面向后续开发、维护和部署人员，说明 `auction-app` 的项目目标、技术架构、目录结构、运行方式、数据模型、API、核心业务计算、导入导出、协同刷新和常见维护流程。

## 1. 项目定位

`auction-app` 是用于替代 `现金会v3.0.xlsx` 的本地局域网协同 Web 应用。它把 Excel 中的拍卖录入、成交确认、精算结算、完整明细导出等流程搬到浏览器中完成。

典型使用方式：

1. 一台电脑作为主机启动服务。
2. 其他电脑、平板或同一局域网内设备通过浏览器访问主机局域网地址。
3. 所有人共享同一份 `data/store.json` 数据。
4. 任意一端新增、修改、删除成交记录后，其他页面通过 SSE 实时收到刷新通知。

## 2. 技术栈

| 层级 | 技术 | 说明 |
|---|---|---|
| 前端 | Vue 3 | 页面状态、表单、表格、结算预览都写在 `src/main.js` |
| 构建 | Vite | 开发服务器、生产构建、API 代理 |
| 后端 | Node.js 原生 HTTP | 不依赖 Express，API 和静态文件服务都在 `server.js` |
| 实时协同 | Server-Sent Events | `/api/events` 向浏览器推送数据变化事件 |
| 存储 | JSON 文件 | 所有业务数据保存到 `data/store.json` |
| Excel 导入 | Python 标准库 | `scripts/import_workbook_data.py` 直接解析 `.xlsx` ZIP/XML |

## 8. 数据文件

主数据文件：

```text
auction-app/data/store.json
```

服务启动时会调用 `loadStore()`：

1. 确保 `data/` 目录存在。
2. 读取 `store.json`。
3. 用 `normalizeStore()` 补齐缺失字段。
4. 如果文件不存在或读取失败，则生成默认数据。

每次需要落盘的接口会调用 `persist()`：

1. 更新 `meta.updatedAt`。
2. 将写入操作追加到 `writeQueue`。
3. 按顺序写回 JSON，降低并发写入互相覆盖的风险。

建议拍卖当天定时复制 `data/store.json` 到备份目录。

## 9. 数据模型

### 9.1 顶层结构

```json
{
  "meta": {},
  "itemCodes": [],
  "sellerCodes": [],
  "customers": [],
  "lots": [],
  "liveEntry": {},
  "audit": []
}
```

| 字段 | 类型 | 说明 |
|---|---|---|
| `meta` | object | 场次名称、开拍时间、默认佣金、更新时间 |
| `itemCodes` | array | 拍品缩写与拍品名称映射 |
| `sellerCodes` | array | 出货英文缩写与日文号牌映射 |
| `customers` | array | 客户、货主、佣金特殊设置 |
| `lots` | array | 成交或待拍记录 |
| `liveEntry` | object | 现场录入草稿，用于数据大屏实时显示 |
| `audit` | array | 最近 250 条操作日志 |

### 9.2 meta

```json
{
  "eventName": "现金拍卖会",
  "startTime": "10:00",
  "sellerCommissionRate": 5,
  "buyerCommissionRate": 10,
  "returnCommissionRate": 5,
  "updatedAt": "2026-05-03T10:05:08.272Z"
}
```

### 9.3 itemCodes

```json
{
  "code": "hn",
  "name": "花鳥"
}
```

`code` 是现场录入时使用的拍品缩写，`name` 是显示在表格、明细和传票上的名称。

### 9.4 sellerCodes

```json
{
  "code": "a",
  "label": "あ"
}
```

`code` 是现场录入时使用的英文缩写，`label` 是出货号牌。系统会通过 `sellerCodes.code -> sellerCodes.label -> customers.sellerLabel` 找到货主。

### 9.5 customers

```json
{
  "id": "uuid",
  "bidderNo": 5,
  "sellerLabel": "あ",
  "name": "客户名称",
  "actualSellerName": "货主实际名称",
  "phone": "电话或备注",
  "sellerRate": "",
  "buyerRate": "",
  "returnRate": ""
}
```

| 字段 | 说明 |
|---|---|
| `bidderNo` | 买家号牌，也是精算按客户汇总的主键 |
| `sellerLabel` | 出货号牌，一个客户可对应多个出货号牌 |
| `name` | 页面主要显示名称 |
| `actualSellerName` | 有值时，精算传票优先显示 |
| `sellerRate` | 特殊出货佣金，空值时使用默认佣金 |
| `buyerRate` | 特殊买货佣金，空值时使用默认佣金 |
| `returnRate` | 预留的退货佣金字段 |

特殊客户：

```json
{
  "bidderNo": -1,
  "name": "流拍"
}
```

买家号牌填 `-1` 时，系统判断为流拍退回。

### 9.6 lots

```json
{
  "id": "uuid",
  "itemNo": 1,
  "sellerCode": "a",
  "itemCode": "hn",
  "quantity": 1,
  "buyerNo": 5,
  "priceK": 8,
  "buyerConfirmed": false,
  "sellerConfirmed": false,
  "returnConfirmed": false,
  "note": "",
  "createdAt": "2026-05-03T10:00:00.000Z",
  "updatedAt": "2026-05-03T10:00:00.000Z"
}
```

| 字段 | 说明 |
|---|---|
| `itemNo` | 拍品编号 |
| `sellerCode` | 出货号牌英文缩写 |
| `itemCode` | 拍品名称缩写 |
| `quantity` | 数量，大于 3 时显示为 `山売` |
| `buyerNo` | 买家号牌；空值代表待拍；`-1` 代表流拍 |
| `priceK` | 千单位成交价，`8` 代表 `8,000` |
| `buyerConfirmed` | 买家收款确认 |
| `sellerConfirmed` | 卖家付款确认 |
| `returnConfirmed` | 退货确认 |
| `note` | 备注 |

## 10. API 说明

所有 API 都由 `server.js` 中的 `routeApi()` 处理。

### 10.1 状态读取

```http
GET /api/state
```

返回完整 `state`。

### 10.2 SSE 事件

```http
GET /api/events
```

返回 `text/event-stream`。连接成功后发送 `hello`，数据变化后广播 `state`。

前端收到 `state` 事件后调用 `/api/state` 重新拉取完整数据。

### 10.3 场次设置

```http
POST /api/meta
Content-Type: application/json
```

请求体是需要合并进 `state.meta` 的字段。

### 10.4 成交记录

| 方法 | 路径 | 说明 |
|---|---|---|
| `POST` | `/api/lots` | 新增成交记录 |
| `PUT` | `/api/lots/:id` | 更新成交记录 |
| `DELETE` | `/api/lots/:id` | 删除成交记录 |
| `POST` | `/api/lots/bulk` | 批量确认或删除 |

批量操作请求体示例：

```json
{
  "ids": ["lot-id-1", "lot-id-2"],
  "action": "buyer",
  "value": true
}
```

`action` 支持：

| action | 说明 |
|---|---|
| `buyer` | 批量切换买家确认 |
| `seller` | 批量切换卖家确认 |
| `return` | 批量切换退货确认 |
| `delete` | 删除所选记录 |

### 10.5 现场草稿

```http
POST /api/live-entry
```

用于把现场录入表单中的草稿同步给其他页面，主要服务于 `数据大屏`。草稿不写入 `store.json`，保存正式拍品后会被清空。

### 10.6 客户资料

| 方法 | 路径 | 说明 |
|---|---|---|
| `POST` | `/api/customers` | 新增或更新客户 |
| `DELETE` | `/api/customers/:id` | 删除客户 |

`POST /api/customers` 如果请求体带有已有 `id`，则更新；否则新增。

### 10.7 代码表

| 方法 | 路径 | 说明 |
|---|---|---|
| `POST` | `/api/codes/items` | 覆盖保存拍品代码 |
| `POST` | `/api/codes/sellers` | 覆盖保存出货号牌代码 |

### 10.8 CSV 导入

```http
POST /api/import/csv
```

请求体：

```json
{
  "csv": "客户号牌,客户名称\n1,张三",
  "onlyCustomers": false
}
```

返回：

```json
{
  "ok": true,
  "customers": 10,
  "lots": 20
}
```

### 10.9 清空本场成交

```http
POST /api/auction/clear
```

只清空 `lots`，保留 `customers`、`itemCodes`、`sellerCodes` 和 `meta`。

### 10.10 导出完整明细

```http
GET /api/export/full.csv
```

返回 UTF-8 BOM CSV，浏览器会下载为 `auction-full.csv`。

## 11. 前端页面模块

页面由 `src/main.js` 中的 Vue 根组件和 `settlement-table` 子组件组成。

| 标签页 | key | 功能 |
|---|---|---|
| 数据大屏 | `screen` | 当前拍品、成交额、佣金、最近成交、排行 |
| 现场录入 | `record` | 快速录入、修改指定拍品、流拍、快速创建拍品代码 |
| 成交登记 | `deals` | 搜索、筛选、排序、批量确认、删除、CSV 导出 |
| 精算结算 | `settlement` | 按客户生成卖方/买方明细和请款/打款额 |
| 客户资料 | `customers` | 新增、修改、删除客户 |
| 场次设置 | `settings` | 场次、佣金、代码表、CSV 导入、清场 |
| 完整明细 | `export` | 类似旧 Excel `完整信息` 的成交汇总 |

## 12. 核心计算逻辑

前后端各有一份 `derive` 逻辑：

| 位置 | 用途 |
|---|---|
| `server.js` 的 `derive(lot)` | 导出 CSV 时计算完整明细 |
| `src/main.js` 的 `deriveLot(lot)` | 页面显示、状态、精算、统计 |

维护时如果修改佣金、税、状态规则，需要同时检查这两处，避免页面显示和 CSV 导出不一致。

### 12.1 关联客户

货主查找：

```text
lot.sellerCode
-> sellerCodes.code
-> sellerCodes.label
-> customers.sellerLabel
```

买家查找：

```text
lot.buyerNo
-> customers.bidderNo
```

### 12.2 成交金额

```text
amount = priceK * 1000
```

例如 `priceK = 8`，成交金额为 `8,000`。

### 12.3 退回判断

| 条件 | returnType |
|---|---|
| 买家号牌等于货主客户号牌 | `顶价退回` |
| 买家号牌等于 `-1` | `流拍退回` |
| 其他 | 空 |

### 12.4 佣金和税

正常成交：

```text
sellerRate = 客户特殊出货佣金 || 默认出货佣金
buyerRate = 客户特殊买货佣金 || 默认买货佣金
sellerCommission = amount * sellerRate / 100
buyerCommission = amount * buyerRate / 100
sellerTax = sellerCommission / 10
buyerTax = buyerCommission / 10
sellerNet = amount - sellerCommission - sellerTax
buyerTotal = amount + buyerCommission + buyerTax
```

流拍：

```text
sellerRate = "NA"
buyerRate = "NA"
sellerCommission = 0
buyerCommission = 0
sellerTax = 0
buyerTax = 0
```

### 12.5 状态规则

| 条件 | 状态 |
|---|---|
| 退回行却勾选买家确认或卖家确认 | `结算异常！` |
| 正常成交却勾选退货确认 | `结算异常！` |
| 买家号牌为空 | `待拍` |
| 退回行且已退货确认 | `已退回` |
| 顶价退回未确认退货 | `顶价待退回` |
| 流拍退回未确认退货 | `流拍待退回` |
| 买家确认且卖家确认 | `已结算` |
| 仅买家确认 | `等待货主结算` |
| 仅卖家确认 | `货主已结算|买家未付款` |
| 正常未确认 | `未结算` |

页面还会把状态映射为买家/卖家动作：

| 状态 | 卖家动作 | 买家动作 |
|---|---|---|
| `未结算` | `待买家` | `可收` |
| `等待货主结算` | `可付` | `已收` |
| `已结算` | `完成` | `完成` |
| `货主已结算|买家未付款` | `待收` | `可收` |
| `顶价待退回` | `顶待退` | `顶待退` |
| `流拍待退回` | `流待退` | `流待退` |
| `已退回` | `已退` | `已退` |

## 13. 精算结算逻辑

精算按 `customers.bidderNo` 选择客户，并同时考虑该客户作为买家和卖家的记录。

卖方明细：

```text
row.sellerNo == 当前客户号牌
```

买方明细：

```text
row.buyerNo == 当前客户号牌
```

默认只显示需要处理的记录：

| 模式 | 默认显示动作 |
|---|---|
| 卖方 | `可付`、`顶待退`、`流待退` |
| 买方 | `可收`、`顶待退`、`流待退` |

过滤项：

| 选项 | 说明 |
|---|---|
| 去除买 | 不计算买方明细 |
| 去除卖 | 不计算卖方明细 |
| 去除退货 | 排除退回记录 |
| 只算退货 | 只显示退回记录 |
| 全部显示 | 不按默认动作过滤 |
| 无视结算状态 | 金额计算时包含全部相关记录 |

结算金额：

```text
买方应收 = 买方明细 buyerTotal 合计
卖方应付 = 卖方明细 sellerNet 合计
最终金额 = 买方应收 - 卖方应付
```

| 最终金额 | 页面含义 |
|---:|---|
| 大于或等于 0 | 请款额 |
| 小于 0 | 打款额 |

## 15. 实时协同机制

服务端维护：

```js
const clients = new Set();
```

浏览器打开页面后连接：

```text
/api/events
```

当 API 修改了持久化数据后：

1. 调用 `persist()` 写入 `store.json`。
2. 调用 `broadcast()` 给所有 SSE 客户端发送 `state` 事件。
3. 前端收到事件后调用 `load()` 重新请求 `/api/state`。

注意：实时协同负责刷新页面数据，不负责字段级冲突合并。多人同时编辑同一条记录时，后保存的一次会覆盖前一次。

## 16. 静态文件服务

生产服务 `serveStatic()` 的文件目录选择：

1. 如果存在 `dist/`，从 `dist/` 读取构建后的文件。
2. 如果不存在 `dist/`，回退到 `public/`。

因此正式使用前建议先执行：

```bash
npm run build
```

## 17. 代码修改注意事项

1. 修改业务计算时，同时检查 `server.js` 的 `derive()` 和 `src/main.js` 的 `deriveLot()`。
2. 修改数据字段时，同时检查 `defaultStore()`、`normalizeStore()`、`cleanLot()`、`cleanCustomer()`、导入脚本和前端表单。
3. 修改 API 路径时，同时检查前端 `api()` 调用和 Vite 代理配置。
4. 修改 CSV 表头识别时，检查 `pick()`、`upsertCustomersFromCsv()` 和 `appendLotsFromCsv()`。
5. 修改精算逻辑时，重点检查 `settlementRowsFor()`、`sellerTotal`、`buyerTotal`、`settlementNet`。
6. 修改页面标签时，检查 `tabs()` 和模板中的 `activeTab` 分支。

http://localhost:4173

发布或交付前建议验证：

2. 新增一条正常成交记录。

4. 在成交登记中切换 `收`、`付`、`退`。
5. 在精算结算中选择客户并检查金额。
6. 导出完整明细 CSV。
7. 备份 `data/store.json`。

## 19. 备份与恢复

### 19.1 备份

关闭服务或在低操作时段复制：

```text
auction-app/data/store.json
```

推荐备份文件名：

```text
store-YYYYMMDD-HHMM.json
```

### 19.2 恢复

1. 停止 `npm start`。
2. 将备份文件复制回 `auction-app/data/store.json`。
3. 重新执行 `npm start`。
4. 打开页面确认数据。

## 20. 常见问题

### 页面打不开

检查：

1. 是否执行了 `npm start`。
2. 端口是否是 `4173`。
3. 其他设备是否和主机在同一局域网。
4. 防火墙是否阻止 Node.js 接入局域网。

### 开发页面 API 失败

开发模式需要同时启动：

```bash
npm start
npm run dev
```

如果只启动 Vite，页面会打开，但 `/api` 无法正常返回数据。

### CSV 导入后客户重复

客户去重条件是：

```text
bidderNo + sellerLabel
```

同一客户多个出货号牌会保留多行，这是正常设计。

### 页面金额和导出金额不一致

优先检查 `src/main.js` 的 `deriveLot()` 与 `server.js` 的 `derive()` 是否同步修改。

### 数据突然变成默认值

检查：

1. `data/store.json` 是否被删除或替换。
2. 当前启动目录是否是正确的 `auction-app`。
3. 是否误执行了 Excel 导入脚本覆盖数据。
